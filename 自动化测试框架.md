#### › 有没有什么好用的自动化测试框架，通过cli 测试网络设备的

• 有，而且这类场景我一般不只看“框架”，而是看“测试框架 + 设备连接库 + 结果报告”这一整套。

  如果你的目标是“通过 CLI 登录网络设备，执行 show/config 命令，再做断言和回归测试”，我比较推荐这 4 套：

  - pyATS + Unicon + Genie
      - 适合：Cisco 为主、网络测试场景比较重、需要 testbed/拓扑/日志/报告/重跑
      - 优点：官方就有 pyats run job、pyats run robot 这类 CLI；--testbed-file、--xunit、清理流程、归档这些都比较完整，偏“正经网络测试平台”
      - 缺点：学习曲线比普通 Python 测试高；如果你不是 Cisco 生态，体验通常没那么顺
      - 资料：
          - pyATS CLI 文档: https://pubhub.devnetcloud.com/media/pyats/docs/cli/pyats_run.html
          - pyATS/Genie/Unicon 官方文档入口: https://developer.cisco.com/docs/pyats/open-source-documentation/
  - Robot Framework + SSHLibrary
      - 适合：想用“关键字驱动”方式写测试，用例希望更像自然语言，团队里有非开发同学
      - 优点：Open Connection、Login、Execute Command 这类关键词很直观，做 SSH CLI 回归很快
      - 缺点：复杂 prompt、分页、交互式确认、切模式时，往往还是要自己封装高层关键字
      - 资料：
          - Robot Framework: https://docs.robotframework.org/
          - SSHLibrary: https://marketsquare.github.io/SSHLibrary/
  - pytest + scrapli / Netmiko
      - 适合：你本身会 Python，想把网络设备测试做成工程化代码，接 CI/CD
      - 优点：pytest 的 fixture/参数化很适合做“多设备、多命令、多断言”；scrapli 对网络设备的 privilege/prompt 处理更现代，Netmiko 平台覆盖广、上手快
      - 缺点：不是开箱即用的“网络测试平台”，很多结构要自己搭
      - 我个人建议：
          - 偏工程化、新项目：pytest + scrapli
          - 想快速上手、平台广：pytest + Netmiko
      - 资料：
          - pytest: https://docs.pytest.org/en/stable/
          - scrapli: https://carlmontanari.github.io/scrapli/user_guide/basic_usage/
          - Netmiko: https://ktbyers.github.io/netmiko/
  - Nornir + NAPALM
      - 适合：设备很多，要并发跑检查，重点是“批量校验/巡检/合规”
      - 优点：Nornir 负责 inventory 和并发；nornir_napalm 有 napalm_cli；NAPALM 还能用 YAML 做状态合规校验，甚至有 CLI 工具
      - 缺点：严格说它更像“编排 + 校验能力”，不是那种完整的测试框架
      - 资料：
          - Nornir: https://github.com/nornir-automation/nornir
          - nornir_napalm: https://nornir.tech/nornir_napalm/html/tutorials/napalm_cli.html
          - NAPALM CLI: https://napalm.readthedocs.io/en/latest/cli.html
          - NAPALM Validate: https://napalm.readthedocs.io/en/latest/validate/

#### › 请给我详细介绍一下   - pytest + scrapli / Netmiko

• 可以，我把这套组合按“是什么、怎么用、优缺点、怎么选、怎么落地”给你讲清楚。

  先给结论：

  - pytest + scrapli / Netmiko 不是一个单独框架，而是一种很常见的网络设备 CLI 自动化测试组合
  - pytest 负责“测试管理”，scrapli 或 Netmiko 负责“连设备、发命令、收结果”
  - 如果你问我怎么选：
      - 新项目、想长期工程化、多厂商但代码要更现代：pytest + scrapli
      - 想快速落地、设备类型杂、遗留设备多：pytest + Netmiko

  下面关于“更适合新项目/更适合遗留设备”的判断，属于我基于官方能力边界做的工程判断，不是官方原话。

  先把角色分清

  - pytest：发现测试、执行测试、做断言、夹具管理、参数化、筛选用例、输出报告
  - scrapli / Netmiko：SSH/Telnet/Serial 连设备，处理 prompt，执行 show / config 命令，返回结果
  - 所以这套组合的本质是：
      - pytest 管“测试生命周期”
      - scrapli/Netmiko 管“设备会话生命周期”

  你可以把它理解成：

  pytest
    -> 按设备/命令生成测试用例
    -> 建立连接 fixture
    -> 调 scrapli/Netmiko 发命令
    -> assert 结果
    -> 输出 junit/xml/终端报告

  为什么 pytest 很适合网络设备测试

  pytest 在这个场景里特别好用，主要是因为 4 个能力：

  - fixture
      - 很适合封装“登录设备 / 退出设备 / 建测试数据 / 清理环境”
      - 官方文档明确支持通过参数名请求 fixture，也支持 yield 做 teardown
  - parametrize
      - 很适合做“同一条测试跑多台设备、多条命令、多组输入”
  - marker
      - 可以把用例分成 smoke、config、destructive、vendor_xxx
      - 运行时用 -m 精确筛选
  - CI 报告
      - pytest 自带 --junitxml，非常适合接 Jenkins / GitLab CI / GitHub Actions

  对于网络设备，这意味着你可以很自然地表达：

  - 同一条检查跑 50 台交换机
  - 只跑 smoke 用例
  - 把“会改配置”的测试和“只读巡检”分开
  - 失败后知道是“哪台设备、哪条命令、哪个断言”挂了

##### pytest + scrapli 详细介绍

  scrapli 的设计更像“现代化的网络 CLI 客户端库”。

  从官方文档能直接看到，它有这些关键特性：

  - 有 5 个核心平台驱动：IOS-XE、NX-OS、IOS-XR、EOS、Junos
  - 非 core 平台可以走 scrapli_community、GenericDriver 或 NetworkDriver
  - 默认 transport 是系统 ssh，也支持 paramiko、ssh2、telnet、asyncssh、asynctelnet
  - 有同步驱动，也有 async 驱动
  - 核心网络驱动理解 privilege level，能自动处理不同模式
  - 返回的是 Response / MultiResponse 对象，不只是裸字符串
  - 支持 TextFSM、Genie、TTP 解析输出
  - 有 send_interactive 处理交互式命令
  - 有 on_open / on_close，很适合“登录后先关分页 / 接受 banner / 切模式”
  - 默认开启严格 host key 检查，这点比很多示例代码更“安全默认”

  你可以把 scrapli 理解成三层：

  - Driver
      - 最底层，比较“原始”，你得自己管更多细节
  - GenericDriver
      - 泛化设备驱动，有 send_command 等便利方法，但不懂你设备的 privilege 语义
  - Core Driver
      - 像 IOSXEDriver、NXOSDriver 这种，最适合官方支持的平台

  对测试工程来说，scrapli 最大的优点是：

  - 抽象层次比较清晰
  - 返回对象信息更丰富，比如耗时、输入命令、输出结果
  - 处理交互、特权级、transport、自定义连接行为更灵活
  - 有 async 路线，后面扩展并发更自然

  它的代价是：

  - 对非官方 core 平台，你可能要用 community driver 或 GenericDriver
  - 学习成本通常比 Netmiko 略高
  - 团队里如果大家只想“赶紧能跑 SSH 命令”，一开始会觉得它没 Netmiko 直接

  一个很小的 pytest + scrapli 例子：

  # conftest.py
  import os
  import pytest
  from scrapli.driver.core import IOSXEDriver

  DEVICES = [
      {"name": "r1", "host": "10.0.0.1"},
      {"name": "r2", "host": "10.0.0.2"},
  ]

  @pytest.fixture(params=DEVICES, ids=lambda d: d["name"])
  def device(request):
      return request.param

  @pytest.fixture(scope="function")
  def conn(device):
      with IOSXEDriver(
          host=device["host"],
          auth_username=os.environ["NET_USER"],
          auth_password=os.environ["NET_PASS"],
          auth_strict_key=False,
      ) as c:
          yield c

  # tests/test_show.py
  def test_show_version(conn):
      resp = conn.send_command("show version")
      assert "Cisco IOS XE" in resp.result

  def test_hostname_exists(conn):
      resp = conn.send_command("show running-config | include ^hostname")
      assert "hostname" in resp.result

  scrapli 这里有几个测试层面很实用的点：

  - resp.result
      - 原始文本结果
  - resp.elapsed_time
      - 可以做简单性能观察
  - resp.textfsm_parse_output() / resp.genie_parse_output()
      - 把 show 输出解析成结构化数据后再断言，通常比直接 assert "up" 稳定

  如果你要做更复杂的连接准备，scrapli 的 on_open 很好用，比如：

  - 登录后先发 terminal length 0
  - 遇到 banner 先确认
  - 某些平台要先切到特定模式

  如果你要跑交互式命令，比如：

  - copy scp: flash:
  - 中间会问用户名、密码、确认

  scrapli 官方提供 send_interactive，这点对网络自动化非常友好。

  pytest + Netmiko 详细介绍

  Netmiko 的思路更像“多厂商 CLI 连接的通用瑞士军刀”。

  官方文档能直接看到这些特性：

  - ConnectHandler(**kwargs) 根据 device_type 选择正确类
  - 目标就是“跨很广的平台”做 show 和 config
  - send_command() 用基于 pattern 的方式读输出，适合大多数 show 命令
  - send_command_timing() 用基于延时的方式读输出，适合 prompt 不稳定的场景
  - send_config_set() 自动进入/退出 config mode，适合批量下发配置
  - send_config_from_file() 可以按文件逐行下发
  - session_log 可以记录完整会话日志
  - 有 SSHDetect 可以自动猜 device_type
  - API 里能看到它支持 ssh、telnet、serial
  - send_command 支持 use_textfsm、use_ttp、use_genie
  - 官方文档明确提示：Netmiko 4.x 里 delay_factor/max_loops 已弃用，建议改用 read_timeout，兼容模式会在 5.x 去掉

##### Netmiko 最大的优点通常是：

  - 上手非常快
  - device_type + ConnectHandler 的心智模型很简单
  - 多厂商/老设备/平台杂的环境里，经常能更快找到现成支持
  - SSHDetect 对“我先连上看看它到底是什么设备”这种场景很有价值
  - session_log 对排查 prompt、分页、命令回显问题特别实用

  它的典型工程风格是：

  - 先把 device_type 确定好
  - 用 ConnectHandler 建连接
  - send_command 跑 show
  - send_config_set 跑配置
  - 遇到特殊场景再调 expect_string、read_timeout、send_command_timing

  一个很小的 pytest + Netmiko 例子：

  # conftest.py
  import os
  import pytest
  from netmiko import ConnectHandler

  DEVICES = [
      {"name": "r1", "host": "10.0.0.1", "device_type": "cisco_ios"},
      {"name": "sw1", "host": "10.0.0.11", "device_type": "cisco_ios"},
  ]

  @pytest.fixture(params=DEVICES, ids=lambda d: d["name"])
  def conn(request):
      d = request.param
      c = ConnectHandler(
          device_type=d["device_type"],
          host=d["host"],
          username=os.environ["NET_USER"],
          password=os.environ["NET_PASS"],
          session_log=f"logs/{d['name']}.log",
      )
      try:
          yield c
      finally:
          c.disconnect()

  # tests/test_show.py
  def test_show_version(conn):
      output = conn.send_command("show version", read_timeout=20)
      assert "Cisco IOS" in output or "Cisco IOS XE" in output

  def test_hostname_exists(conn):
      output = conn.send_command(
          "show running-config | include ^hostname",
          read_timeout=20,
      )
      assert "hostname" in output

  如果你想直接拿结构化结果，可以这样思路：

  rows = conn.send_command(
      "show ip int brief",
      use_textfsm=True,
      read_timeout=20,
  )
  assert rows

  对 Netmiko，测试里最重要的几个参数通常是：

  - read_timeout
      - 现在更推荐直接用它来控制等待
  - expect_string
      - 如果设备 prompt 或结束标志比较特殊，显式指定更稳
  - session_log
      - 出问题先看日志，往往马上就知道是分页、回显、banner 还是 prompt
  - send_command_timing
      - 遇到交互不稳定、pattern 不好写时可以试，但要谨慎控制超时和后续状态

  scrapli 和 Netmiko 的核心区别

  你可以这样记：

  - scrapli
      - 更像“现代、分层清晰、可扩展的网络 CLI SDK”
  - Netmiko
      - 更像“平台覆盖广、落地快、实战经验很丰富的 CLI 工具库”

  如果按实际选型看，我会这样建议：

  - 选 scrapli，如果你更在意：
      - 新项目
      - 代码结构
      - async 能力
      - transport 灵活性
      - 交互和 privilege 建模
      - 更明确的 Response 对象
  - 选 Netmiko，如果你更在意：
      - 多厂商平台覆盖
      - 遗留设备
      - 快速上手
      - 直接 device_type -> ConnectHandler
      - 设备类型自动探测
      - 团队里已经很多人熟悉 Netmiko

  还有一个很实际的判断：

  - 如果你常测的是 Cisco / Arista / Juniper 这类 scrapli core driver 直接覆盖的平台，scrapli 会很顺
  - 如果你常测的是更杂的设备集合，尤其很多非 core 平台，Netmiko 往往更省事
      - 这是我根据官方文档里 scrapli core 平台范围，以及 Netmiko“broad set of platforms”/device_type/SSHDetect 能力做出的工程判断

  在测试工程里该怎么组织

  我比较推荐这样的目录：

  project/
    inventory/
      devices.yaml
    tests/
      test_smoke.py
      test_config.py
    conftest.py
    pytest.ini

  pytest.ini 可以至少做两件事：

  - 注册 markers
  - 配默认参数

  例如：

  [pytest]
  markers =
      smoke: 只读基础检查
      config: 会下发配置的测试
      destructive: 有破坏性的测试
  addopts = -q --strict-markers

  运行时就可以这样：

  pytest -m smoke --junitxml=reports/junit.xml

  这对网络设备特别重要，因为你一定要把测试分层：

  - smoke
      - 登录、版本、时间、主机名、关键接口状态
  - compliance
      - NTP、SNMP、AAA、VLAN、trunk、路由邻居等配置/状态校验
  - config
      - 会改配置的测试
  - destructive
      - reboot、切主备、接口 flap、删除配置之类，只能在实验环境跑

  真正落地时的最佳实践

  - 连接和测试分离
      - 设备信息放 inventory，不要写死在测试代码里
  - 密码不要写代码里
      - 用环境变量、密钥、vault
  - 只读测试和改配置测试分开
      - 不然很容易把回归测试跑成事故
  - 尽量断言结构化数据，不要过度依赖原始文本
      - 文本输出会因为空格、顺序、版本变化变脆
  - 每台设备保留会话日志
      - 尤其用 Netmiko 时，session_log 很值
  - fixture scope 要想清楚
      - function：最干净，隔离好，但连接开销大
      - module/session：更快，但要注意测试间污染
  - 配置类测试要有回滚思路
      - pre-check -> apply -> verify -> rollback -> verify
  - 对 Netmiko，优先用 read_timeout
      - 官方已经明确 4.x 弃用旧的 delay_factor/max_loops 方案
  - 对 scrapli，非 core 平台先判断
      - 是不是 community driver 能直接用；不行再上 GenericDriver

  如果你现在就要选，我会这样给建议

  - 你是“Python 写得比较顺，想把网络测试做成长期工程”：
      - 选 pytest + scrapli
  - 你是“设备非常杂，先求能跑起来、能覆盖更多老平台”：
      - 选 pytest + Netmiko
  - 你是“只做读命令巡检，不太碰复杂交互”：
      - 两个都行，Netmiko 会更直接
  - 你是“后面可能要做复杂交互、异步、transport 定制、长线维护”：
      - 更偏 scrapli
